# Day 0

우연히 링크드인을 훑어보다가 좋은 글을 읽었습니다.

> "지금부터 라이브러리를 만들고, 프레임워크를 만드세요.  
> 10년 후에는 AI가 여러분을 대체할 겁니다."

당시에는 회사에서 여러 프로젝트들에 치이던 중이라 조용히 '좋아요'만 누르고 지나쳤는데,  
계속 머리를 맴도는 것을 보니 내심 위기감을 느끼고 있었나봅니다.

자려고 누워서 최근 개발하는 제 모습을 반추해보니,  
Cursor나 Claude code등 수많은 AI 도구들이 우후죽순 등장하고 회사에서도 공식적으로 사용을 승인해주면서 스스로 깊게 고민하는 시간이 줄었더라구요.  
개발을 처음 시작하던 시기에 비하면 몹시도 관성적으로 개발을 하고 있었습니다. 

그러다 문득 학부 시절에 들었던 `CS330`이라는 악명 높았던 운영체제 강의가 생각이 났습니다.
[pintos](https://oslab.kaist.ac.kr/ee415-spring-2022-pintos-introduction/)라는 c언어로 작성된 작지만 매콤한 운영체제 프레임워크에 스케쥴링, 파일 시스템, 가상 메모리 등을 직접 구현하는 프로젝트였습니다.  
정말 지독한(?) 프로젝트였습니다. 커널 기능 하나 만들려면 메모리부터 malloc, dealloc도 해주어야 하구요. 뭐 하나 쉽게 되는게 없었습니다. 
당시에는 ChatGPT도 없었으니 나는 컴퓨터다 컴퓨터다 중얼거리면서 매일 같이 밤을 새기 일쑤였습니다.

그래서인지 전산학과 학생들이 가장 기피했던 과목이기도 했고, 과제 마감 전날에는 디버깅은 포기하고 테스트 통과를 바라며 기도하던 기억이 납니다

그 당시에는 정말 고되긴 했지만 여튼 종강 즈음 되어서 우리가 사용하는 현대 운영체제도 이런 비스무리한 추상화가 되어 있겠거니 하는 작은 배움을 얻을 수 있었습니다.

그에 반해 요새는 제가 하고 있는 개발은 참 간편합니다.

```tsx
const SimpleButton = () => {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount((prev) => prev + 1)}>like {count}</button>
  );
};
```

프론트엔드 개발을 아예 모르는 사람이 슬쩍 읽어보아도, 대충 누르면 +1 이 되는 버튼이겠구나 라고 이해가 됩니다.  
리렌더링, 이벤트 핸들링, 최적화, 상태 관리 같은 복잡한 동작은 프레임워크가 모두 알아서 처리해주고 있으니까요.


토스의 [Frontend Fundamentals](https://frontend-fundamentals.com/code-quality/code/examples/login-start-page.html#%F0%9F%94%8D-%E1%84%83%E1%85%A5-%E1%84%8B%E1%85%A1%E1%86%AF%E1%84%8B%E1%85%A1%E1%84%87%E1%85%A9%E1%84%80%E1%85%B5-%E1%84%8E%E1%85%AE%E1%84%89%E1%85%A1%E1%86%BC%E1%84%92%E1%85%AA)를 읽다가 제 생각을 완벽하게 표현한 문장이 있어 인용해봅니다.

> "왼쪽으로 10걸음 걸어라" 라고 하는 문장이 있어요. 여기에서
>
> “왼쪽”은 “북쪽을 바라보았을 때 90도 돌아간 위치” 를 추상화한 것이고,  
> “90도”는 “한 번의 회전을 360등분한 각의 90배만큼 시초선에 대해 시계 반대 방향으로 돌아간 것” 을 추상화한 것이고,  
> “시계 방향” 의 정의는 “북반구에서 해시계의 바늘이 돌아가는 방향” 을 추상화한 것이에요.

React라는 훌륭한 프레임워크가 메인스트림이 된 이후에 개발을 시작해서 그럴지도 모르겠습니다.
이제 매일 사용하는 React를 쓰다보면 useState든 useMemo든 reconciliation 이든 동작 원리에 대해서는 크게 고민을 안합니다.

그래서 26년 새해를 맞이하며 scratch부터 프레임워크 비스무리한 것을 만들어보려 합니다.

pintos의 어원을 찾아보니 멕시코에서 자주 먹는 핀토콩에서 출발했다고 하더라구요.

비록 저는 좋아하지 않습니다만 비슷하게 멕시코에서 자주 먹는 고수(cilantro) 로 프레임워크 이름을 정해보았습니다.

아마 시행착오도 많고 읽으시면서 답답하실 수도 있겠지만, 조그만 인사이트가 되길 바라면서 시작해보겠습니다
