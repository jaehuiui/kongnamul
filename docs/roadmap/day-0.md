# Day 0: 프롤로그

여러분은 왜 React를 사용하시나요?

- 프론트엔드 생태계에서 React의 비중이 압도적이라 그런가요
- React의 코드 철학과 아키텍쳐의 열렬한 팬이신가요
- React가 여러분이 재직 중인 회사의 표준 스택이라는 현실적인 관점도 있겠습니다

저는 스타트업을 창업하며 무작정 프론트엔드 개발을 시작했는데,  
많은 회사에서 사용한다는 말을 듣고 React를 택했고 지금까지도 관성적으로 사용하고 있습니다.

React가 프론트엔드 생태계에서 아직까지 굳건하게 자리를 지켜주고 있어서 다행이고 고마울 따름입니다.

---

### [1] 관성은 무섭습니다

마침 **관성**이라는 표현이 나왔으니 조금 더 이야기를 이어 가보자면,

개발을 할 때마다 꽤나 관성적으로 코드를 작성하고 있다는 생각이 문득 들곤 합니다.

습관적으로 React 컴포넌트를 만들고, state를 추가하고, API 요청을 포함한 비즈니스 로직을 포함한 hook을 추가합니다.  
그렇게 기존에 구현된 코드와 이질감이 없는, 왠지 모르게 이미 있었던 것만 같은 코드를 주로 작성하려고 노력합니다.

낯선 도메인에서 새로운 서비스 개발에 착수할 때도 비슷합니다.  
결국에는 React를 비롯한, 검증되고 익숙한 도구들을 조합해 '**안전한 선택**'을 합니다.

<img src="../assets/day-0_1.png" width="500">

그래서인지 매년 발표되는 [State of JavaScript](https://2024.stateofjs.com/en-US/libraries/front-end-frameworks/)를 보면 새로운 라이브러리와 프레임워크가 계속 등장함에도 React는 여전히 굳건합니다.

기술적으로 우수하다는 이유도 있겠지만  
저처럼 관성을 유지하려는 개발자들의 선택도 큰 비중을 차지하지 않았나 생각합니다.

더군다나 AI의 등장으로 인해 이 관성은 걷잡을 수 없이 강해졌다고 느껴집니다.

이제는 어떤 기술이 가진 독창적인 철학과 내부 구현을 궁금해하기보단,  
그 기술을 실제로 활용하는 방법과 어떤 결과물을 만들 수 있는지에 집중하는 세상으로 변하고 있습니다.  
더 이상 AI에게 학습을 시키기보단,  
되려 개발자가 AI가 작성한 패턴을 학습하는 모습을 심심치 않게 볼 수 있습니다.

물론 AI는 참 똑똑합니다. 편하기도 하구요.

편안함에 잠식되다 보면 도태될 것만 같은 무서운 예감이 듭니다.

---

### [2] React가 없는 세상이라면

무서운 예감이 현실이 된다면 어떨까요.

마치 실수로 게임의 세이브 파일이 지워진 것처럼  
어느 날 자고 일어나니 React가 없는 세상이 된거죠.

이왕이면 React만 없어진 게 아니라 JS 기반의 모든 UI 라이브러리와 프레임워크가 전부 사라진 세상인 겁니다.

관성처럼 개발하던 모든 코드 스니펫이 무용지물이 되고, 심지어 AI도 아무런 도움을 줄 수 없습니다.  
허탈한 마음이 들겠지만 그래도 우리는 일을 해야만 합니다.

당장 바닥부터 UI 라이브러리나 프레임워크를 만들어야 할 것입니다.

여러분은 어떤 도구를 만드실 건가요?

세상에서 사라진 React를 상상하며 다시 만들어볼 수도 있고,  
개념적으로 알고는 있지만 사용해 보지는 못했던 색다른 개념을 더해볼 수도 있습니다.

- React와 유사한 Virtual DOM 개념을 사용하고 런타임 환경에서 동작할 수도 있고
- Svelte처럼 컴파일 단계에서 순수한 JS 코드 결과가 생성될 수도 있고
- Solid처럼 signal 기반으로 DOM 업데이트를 트리거할 수도 있습니다

막상 생각해 보니 막막합니다.

매일같이 React를 사용해 왔지만  
React의 설계 과정에서 어떤 고민과 선택을 했을지는 깊게 고민해본 적이 없어서 그런 걸까요.

**역설적이지만 React를 이해하려면 React가 사라져야 하지 않을까**라는 생각을 전하고 싶었습니다.

### [3] 무엇을 만들어볼까요

링크드인에서 우연히 어떤 포스트를 읽었습니다.

> "지금부터 라이브러리를 만들고, 프레임워크를 만드세요.  
> 10년 후에는 AI가 여러분을 대체할 겁니다."

앞서 관성 이야기에서도 언급했던 것처럼 도태된다는 사실은 두렵습니다.

그렇다면 왜 라이브러리나 프레임워크를 만들어야할까요?

라이브러리나 프레임워크를 만든다는 건  
결국 **문제를 정의하고, 선택의 이유를 고민**하는 일입니다.

AI는 이미 존재하는 결과물과 패턴을 학습하는 데는 탁월하지만,

결과물이 나올 때까지의 수많은 과정을 **고민**하고,  
세상에 없는 무언가를 새롭게 **정의**하고,  
낯선 분야를 **탐구**하는 것만큼은, 적어도 아직까지는,  
인간이 잘할 수 있는 영역이라 믿고 있습니다.

그래서 React가 사라진 세상에서  
웹 어플리케이션을 구현할 수 있는 도구를 바닥부터 만들어보려고 합니다.

우리에게 익숙한 JSX나 Virtual DOM과 같은 개념들을 당연하게 건너뛰지 않고,  
상태 관리, 렌더링, 업데이트 전략, 불변성과 같은 수많은 기술적 고민들과 trade-off를 기록하고 어떤 의사결정을 내렸는지 기록해보려합니다.

React가 사라진 세상의 생존기처럼 글도 작성하면서 매일 작성한 코드도 커밋 단위로 남겨보고자 합니다.

### [4] Do you know Kongnamul?

제일 중요한 과정이 남았습니다.  
바로 프로젝트의 이름을 정할 차례입니다.

우습게도 문득 대학에서 수강했던 '**[CS330] 운영체제 및 실험**' 이라는 강의가 떠올랐습니다.

[pintos](https://casys-kaist.github.io/pintos-kaist/)라는 C언어로 작성된 운영체제 프레임워크를 사용했는데   
말 그대로 '실행만 되는 수준'의 뼈대만 제공되고 스레드와 스케쥴링, 시스템 콜, 파일 시스템, 가상 메모리 등을 직접 구현하는 프로젝트였습니다.

모든 순간이 힘들고 수십 번도 그만두고 싶었지만  
아이러니하게도 처음으로 코딩을 하면서 묘한 쾌감을 느낀 경험이었습니다.  

며칠씩 고민해서 **설계**한 내용을 코드로 옮겨보았고 고된 과정이었지만  
삐걱거리더라도 **돌아가는 무언가**를 만들었던 것이니까요.

이번 프로젝트도 왠지 비슷한 느낌이 듭니다.

pintos의 어원을 찾아보니 운영체제를 의미하는 OS의 pintOS가 아닌,  
멕시코에서 자주 먹는 핀토콩(pinto)에서 유래했다고 하더라구요.

이왕이면 콩이랑 비슷한 무언가면 좋겠습니다.  
완두콩도 있고 콩으로 만든 두부도 있겠죠.

찾아보니 **콩나물**은 전 세계에서 거의 한국에서만 먹는 식재료라고 하네요.  
그래서 이번 프로젝트의 이름은 `kongnamul`입니다.

<img src="../../assets/icon.png" width="320">

이번 콩나물 시리즈는 **기술적 의사결정을 기록하는 여정**입니다.

진짜 콩나물처럼 하루가 다르게 쑥쑥 크면 좋겠지만,  
시행착오도 있고 쉬운 길도 돌아가는 것처럼 보일 수도 있을 것 같습니다.

그럼에도 불구하고 이 기록이 조그만 인사이트가 되고  
매일 자려고 누워 생각나는 고민 하나쯤은 남길 수 있기를 바라며 여정을 시작해보겠습니다.

감사합니다.
